我想要做一个类似的产品，悬浮窗具有Apple灵动岛的感觉。能够自动检测当前windows系统+WSL下活跃的Codex进程，然后根据进程数量显示相应的呼吸灯，呼吸灯状态表示（蓝色=正在运行、绿色=完成、黄色=需要人工接入、红色=报错中止）。整个状态跟踪与活跃会话的跟踪可以使用Codex原生的hook能力，创建/编辑对应的状态文件，悬浮窗只需要读取状态文件并显示相应效果。


你的方案方向是对的，而且比“直接扫描进程状态”靠谱很多。这个产品可以做成：

> **Codex 状态采集层：hook / wrapper 写状态文件**
> **桌面展示层：Windows 悬浮窗读取状态文件并渲染灵动岛 UI**
> **兜底层：Windows + WSL 进程扫描，清理异常/过期会话**

Codex hooks 目前已经支持 `SessionStart`、`UserPromptSubmit`、`PermissionRequest`、`PreToolUse`、`PostToolUse`、`Stop` 等事件；每个命令型 hook 都会通过标准输入收到一个 JSON 对象，里面包含 `session_id`、`cwd`、`hook_event_name`、`model` 等公共字段，部分事件还包含 `turn_id`。这正好可以作为你的状态数据来源。([OpenAI 开发者][1])

## 1. 产品定位可以叫：Codex Island

不是“监控软件”，而是：

**一个 Codex 多任务状态岛。**

核心展示：

```text
蓝色呼吸灯：Codex 正在执行
绿色呼吸灯：上一轮任务完成 / 当前会话空闲
黄色呼吸灯：需要人工介入，比如权限审批、确认、输入
红色呼吸灯：异常中止、进程消失、hook 判断失败
灰色呼吸灯：会话过期 / 无响应 / 已归档
```

“呼吸灯”就是一个小圆点或胶囊灯，通过透明度、缩放、阴影做周期动画，让用户在视觉上感知状态变化。

## 2. 最稳的架构

```text
┌──────────────────────────┐
│ Windows 桌面悬浮窗        │
│ Tauri / Electron / WinUI  │
│ 读取 status.json          │
└─────────────▲────────────┘
              │
              │ 文件监听
              │
┌─────────────┴────────────┐
│ 状态文件目录              │
│ %LOCALAPPDATA%/CodexIsland │
│ sessions/*.json           │
└─────────────▲────────────┘
              │
              │ 写入
              │
┌─────────────┴────────────┐
│ Codex hooks               │
│ SessionStart              │
│ UserPromptSubmit          │
│ PermissionRequest         │
│ PreToolUse / PostToolUse  │
│ Stop                      │
└─────────────▲────────────┘
              │
              │ 兜底
              │
┌─────────────┴────────────┐
│ 进程扫描器                │
│ Windows: codex.exe        │
│ WSL: pgrep / ps           │
└──────────────────────────┘
```

这里的 **hook** 可以理解为“事件钩子”：Codex 运行到某些关键节点时，自动调用你写的脚本。
**wrapper** 可以理解为“包装启动器”：用户仍然输入 `codex`，但实际上先经过你的启动脚本，用来记录进程、环境变量、状态目录等信息。

## 3. 状态不要只用一个字段，建议拆成三层

你想展示的是“蓝绿黄红”，但内部数据最好更细：

```json
{
  "session_id": "abc123",
  "turn_id": "turn-456",
  "source": "wsl",
  "distro": "Ubuntu-24.04",
  "pid": 12345,
  "cwd": "/home/zkai/project/web3",
  "model": "gpt-5.5-codex",
  "process_state": "alive",
  "turn_state": "running",
  "ui_state": "blue",
  "last_event": "PreToolUse",
  "last_tool": "Bash",
  "last_message": "Running tests",
  "started_at": "2026-06-03T10:21:00-04:00",
  "updated_at": "2026-06-03T10:23:18-04:00"
}
```

这样后面扩展会很舒服。

### 推荐状态映射

| Codex 事件             | 内部状态              | UI 颜色  | 含义                         |
| -------------------- | ----------------- | ------ | -------------------------- |
| `SessionStart`       | `session_started` | 蓝色     | 会话已启动                      |
| `UserPromptSubmit`   | `running`         | 蓝色     | 用户提交了新任务，Codex 开始处理        |
| `PreToolUse`         | `running_tool`    | 蓝色     | 准备调用工具，比如 Bash、编辑文件、MCP 工具 |
| `PostToolUse` 成功     | `running`         | 蓝色     | 工具执行完成，Codex 继续思考          |
| `PermissionRequest`  | `needs_attention` | 黄色     | 需要人工批准权限                   |
| `Stop`               | `completed`       | 绿色     | 当前轮完成，等待下一条用户输入            |
| 进程退出且最后状态为 completed | `closed`          | 灰色/隐藏  | 正常结束                       |
| 进程退出但没有 Stop         | `crashed`         | 红色     | 异常中止                       |
| 状态超过 N 分钟没更新         | `stale`           | 红色或灰色  | 可能卡死或 WSL 断连               |
| `PostToolUse` 返回非零状态 | `tool_warning`    | 可短暂黄/红 | 命令失败，但不一定代表整个任务失败          |

注意：`PostToolUse` 会在 Bash 命令非零退出后触发，但“测试失败”有时是 Codex 正在修复问题的正常过程，不一定应该立刻显示红色。官方文档也说明 `PostToolUse` 是工具执行之后触发，包含 Bash、`apply_patch` 和 MCP 工具，且 Bash 非零退出也会触发。([OpenAI 开发者][1])

所以我建议：

```text
单次命令失败：短暂黄色闪烁
最终 Stop 且失败信息明显：红色
进程异常退出：红色
长期无心跳：红色或灰色
```

## 4. 状态文件设计

不要所有会话写一个大 JSON，建议：

```text
%LOCALAPPDATA%/CodexIsland/
  sessions/
    <session_id>.json
  index.json
  logs/
```

每个 Codex 会话一个文件，避免多个 Codex 并发写同一个文件造成冲突。

例如：

```text
C:\Users\zk\AppData\Local\CodexIsland\sessions\abc123.json
```

WSL 侧可以写入：

```bash
/mnt/c/Users/zk/AppData/Local/CodexIsland/sessions/abc123.json
```

Windows 可以访问 WSL 文件系统，WSL 也可以访问 Windows 文件系统；微软文档也说明可以在 Windows 命令行调用 Linux 命令，或者在 WSL 中访问 Windows 挂载路径。([Microsoft Learn][2])
但为了 Windows 悬浮窗稳定监听文件变化，我更建议把状态目录放在 Windows 的 `%LOCALAPPDATA%`，而不是放在 `\\wsl$` 下面。`\\wsl$` 依赖 WSL 的 9P 文件服务，出问题时 Windows 侧可能访问不到。([Microsoft Learn][3])

## 5. hook 配置示例

全局安装在：

```text
~/.codex/hooks.json
```

示例：

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume",
        "hooks": [
          {
            "type": "command",
            "command": "python3 ~/.codex/codex-island/hook.py",
            "timeout": 5,
            "statusMessage": "Updating Codex Island"
          }
        ]
      }
    ],
    "UserPromptSubmit": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "python3 ~/.codex/codex-island/hook.py",
            "timeout": 5
          }
        ]
      }
    ],
    "PermissionRequest": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "python3 ~/.codex/codex-island/hook.py",
            "timeout": 5
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "python3 ~/.codex/codex-island/hook.py",
            "timeout": 5
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "python3 ~/.codex/codex-island/hook.py",
            "timeout": 5
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "python3 ~/.codex/codex-island/hook.py",
            "timeout": 5
          }
        ]
      }
    ]
  }
}
```

Codex 会从 `~/.codex/hooks.json`、`~/.codex/config.toml`、项目内 `.codex/hooks.json`、项目内 `.codex/config.toml` 等位置加载 hooks；非托管 hook 第一次运行前需要用户用 `/hooks` 审核并信任。([OpenAI 开发者][1])

## 6. hook.py 的核心逻辑

伪代码大概这样：

```python
import json
import os
import sys
import time
from pathlib import Path

payload = json.load(sys.stdin)

event = payload.get("hook_event_name")
session_id = payload.get("session_id")
turn_id = payload.get("turn_id")
cwd = payload.get("cwd")
model = payload.get("model")
tool_name = payload.get("tool_name")
permission_mode = payload.get("permission_mode")

state_dir = os.environ.get(
    "CODEX_ISLAND_STATE_DIR",
    "/mnt/c/Users/zk/AppData/Local/CodexIsland/sessions"
)

Path(state_dir).mkdir(parents=True, exist_ok=True)
state_file = Path(state_dir) / f"{session_id}.json"

def ui_state_from_event(event):
    if event == "PermissionRequest":
        return "yellow"
    if event == "Stop":
        return "green"
    if event in ["SessionStart", "UserPromptSubmit", "PreToolUse", "PostToolUse"]:
        return "blue"
    return "blue"

state = {
    "session_id": session_id,
    "turn_id": turn_id,
    "cwd": cwd,
    "model": model,
    "source": "wsl" if "WSL_DISTRO_NAME" in os.environ else "windows",
    "distro": os.environ.get("WSL_DISTRO_NAME"),
    "pid": os.getppid(),
    "last_event": event,
    "last_tool": tool_name,
    "permission_mode": permission_mode,
    "ui_state": ui_state_from_event(event),
    "updated_at": time.time()
}

tmp = state_file.with_suffix(".tmp")
tmp.write_text(json.dumps(state, ensure_ascii=False, indent=2), encoding="utf-8")
tmp.replace(state_file)
```

这里用 `tmp.replace(state_file)` 是为了做**原子写入**。
“原子写入”意思是：先写临时文件，再一次性替换目标文件，避免悬浮窗读到半截 JSON。

## 7. 为什么还需要进程扫描

因为 hook 不是 100% 完整遥测接口。

官方文档里也明确提到，`PreToolUse` / `PostToolUse` 目前并不能拦截所有 shell 调用，较新的 `unified_exec` 机制也还存在拦截不完整的问题。([OpenAI 开发者][1])
另外 GitHub 上也有人反馈 Linux/Windows 上大文件编辑时 hook 可能因为参数长度问题失效；这说明产品设计不能完全依赖 hook，必须有兜底。([GitHub][4])

进程扫描器负责：

```text
1. 定期扫描 Windows 原生 codex 进程
2. 定期扫描 WSL 内 codex 进程
3. 对比 sessions/*.json
4. 如果状态文件存在但进程不存在，标记 closed/crashed
5. 如果进程存在但没有状态文件，标记 unknown
6. 如果状态超过 N 分钟没更新，标记 stale
```

WSL 侧可以用：

```bash
pgrep -af codex
```

Windows 侧可以用：

```powershell
Get-Process | Where-Object { $_.ProcessName -like "*codex*" }
```

更产品化一点，Windows 悬浮窗主进程可以周期性调用：

```powershell
wsl.exe -d Ubuntu-24.04 -- pgrep -af codex
```

Codex CLI 本身支持在 Windows 原生 PowerShell 下运行，也支持在 WSL2 里运行。([OpenAI 开发者][5])

## 8. 悬浮窗技术栈建议

我建议优先用 **Tauri**。

| 技术栈           | 适合度 | 解释                                            |
| ------------- | --: | --------------------------------------------- |
| **Tauri**     |  最高 | 用系统 WebView 做 UI，Rust 做底层，体积小、内存占用低，适合常驻桌面小工具 |
| Electron      |  中高 | 开发快，CSS 动画方便，但内存占用偏高                          |
| WinUI 3 / WPF |   中 | Windows 原生体验好，但做灵动岛动画和跨平台生态不如前端顺手             |
| Rainmeter     | 低到中 | 做原型快，但产品化能力弱                                  |
| Flutter       |   中 | UI 能力强，但 Windows 桌面小悬浮窗细节要额外处理                |

Tauri 的优势是：
**透明窗口、置顶窗口、托盘图标、文件监听、开机自启、低资源占用**都比较适合这个项目。

UI 可以做成两种状态：

```text
收起态：
[ ● ● ● ]  只显示多个呼吸灯

展开态：
Codex Island
● web3-agent-refactor   Running
● wsl-proxy-fix         Need approval
● codex-sdk-test        Done
```

点击某个任务可以展开详情：

```text
项目路径：/home/zkai/project/web3
模型：gpt-5.5-codex
状态：正在执行 Bash
最后事件：PreToolUse
最后更新：12 秒前
```

## 9. 关键产品细节

### 多 Codex 并行任务显示

建议规则：

```text
1 个任务：一个大呼吸灯
2-4 个任务：多个小呼吸灯横排
5 个以上：显示前 4 个 + “+N”
```

颜色优先级：

```text
红色 > 黄色 > 蓝色 > 绿色 > 灰色
```

也就是只要有一个任务需要人工介入，整个灵动岛外圈可以变成黄色；只要有一个异常，整体外圈变红。

### 状态过期策略

```text
蓝色状态超过 10 分钟无更新：变成灰色或红色
绿色状态保留 5-10 分钟：然后自动折叠
红色状态保留直到用户点击清除
黄色状态一直保留，直到有新的 PreToolUse / PostToolUse / Stop
```

### 不要默认记录 prompt 全文

`UserPromptSubmit` 事件会包含用户提交的 prompt。官方文档显示它的字段里包含 `prompt`。([OpenAI 开发者][1])
但产品上不建议默认写入状态文件，因为里面可能有密钥、代码、隐私信息。

建议默认只记录：

```json
{
  "prompt_preview": null,
  "prompt_hash": "sha256..."
}
```

给用户一个设置：

```text
是否显示任务标题：
- 不显示
- 显示当前目录名
- 显示 prompt 前 30 字
- 自定义别名
```

## 10. MVP 开发路线

### 第一版：只做 WSL + 状态文件 + 悬浮窗

功能：

```text
1. 安装 ~/.codex/hooks.json
2. hook.py 写 sessions/*.json
3. Tauri 悬浮窗读取 JSON
4. 显示蓝/绿/黄/红呼吸灯
5. 点击展开任务列表
```

先不要做复杂进程扫描。

### 第二版：加进程扫描兜底

功能：

```text
1. 扫描 Windows codex 进程
2. 扫描 WSL codex 进程
3. 自动清理 stale session
4. 识别 crashed / unknown
```

### 第三版：产品化

功能：

```text
1. 开机自启
2. 托盘菜单
3. 一键安装/卸载 Codex hooks
4. 自定义状态目录
5. 多 WSL 发行版支持
6. 任务别名
7. 通知声音
8. 任务完成 Windows Toast 通知
```

Windows Toast 通知就是系统右下角通知。

## 11. 我对这个产品的判断

这个产品是有价值的，尤其适合你现在这种：

```text
Windows 主系统
WSL 里跑多个 Codex
Cursor / Terminal 多窗口并行
经常需要知道哪个任务完成、哪个任务卡住、哪个任务需要确认
```

它的核心差异不是“系统监控”，而是：

> **把多个 Codex Agent 的执行状态变成一个低干扰、常驻、可感知的桌面状态层。**

我建议你不要一开始就做复杂 Agent 管理平台，而是先做一个非常轻的：

```text
Codex Island = hook 状态采集器 + 桌面灵动岛悬浮窗
```

这个切口小、需求明确、实现可控，而且后面可以自然扩展成：

```text
Codex 多任务看板
Codex 通知中心
Codex 会话历史
Codex 任务恢复入口
Codex 执行日志聚合
```

真正要注意的是：**hook 负责状态，进程扫描负责兜底，悬浮窗只负责展示**。这三层不要混在一起。
